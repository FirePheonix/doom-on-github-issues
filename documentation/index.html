Release docs

v4 reference

Release shape

v4 turns the project from a replay-heavy prototype into a faster DoomGeneric-first runtime. One GitHub comment should produce one final frame quickly, without replaying the full issue history when a live session is already available.

What changed

  • Cached startup and menu frames make common early states effectively instant.
  • Shared cached menu images use deterministic S3 object keys instead of Redis indirection.
  • The hot path is now parse, update state, render or cache hit, publish, and patch the issue.
  • Multiline and compact commands batch into one final frame update per comment.

Runtime improvements

  • The DoomGeneric worker is persistent when DOOM_ENGINE=doomgeneric.
  • The native C worker creates once and ticks continuously.
  • The Python bridge ignores banner noise and waits for protocol replies.
  • Background sync coalesces to the latest target instead of draining stale work.

Known boundaries

  • GitHub issue patch latency still exists outside the app.
  • GitHub image refresh and proxy latency still exist outside the app.
  • S3 upload time still matters for uncached live frames.
  • Worker health still matters even with the cleaner runtime split.

Architecture

GitHub stays both the controller and the display surface. Webhooks enqueue issue work, per-issue lifecycle code applies commands, cached states short-circuit early menu frames, and a persistent DoomGeneric worker produces live frames when the session is warm.

Core components

  • routes/webhook.js verifies webhook traffic and returns quickly.
  • jobs/queue.js keeps render work off the synchronous webhook path.
  • jobs/locks.js serializes work per issue.
  • sessions/lifecycle.js applies commands and updates the issue body.
  • sessions/artifacts.js chooses cache hit, persistent render, or replay fallback.

State and storage

  • Session state tracks issue history and live sync metadata.
  • Live frame PNGs can publish locally or to S3.
  • Shared cached menu PNGs are keyed deterministically.
  • The issue body always points to the latest visible frame.
  • Background sync keeps the live session near the user-visible history.

Runtime flows

Cached menu frame

The webhook is accepted, commands are parsed, the menu cache resolves a known history, and the issue body is patched using the shared cached URL while the live session primes in the background.

Live persistent frame

The lifecycle applies only the new command batch, the engine sends a step request to the Python worker, the DoomGeneric worker ticks forward, the PNG is published, and the issue body is patched once.

Replay fallback

If the persistent worker is not ready, the visible path falls back to replay rendering from session JSON, publishes the frame, and keeps priming the live session for later comments.

Persistent DoomGeneric

Why it matters

Replay rendering gets slower as history grows because every later frame replays from tick zero. A live DoomGeneric worker keeps game state in memory, so later comments only apply the missing suffix.

Worker protocol

  • READY when the native worker is live.
  • STEP to apply a command batch and capture a frame.
  • SNAPSHOT to capture without new input.
  • SHUTDOWN to stop the worker cleanly.

Startup timeouts are split on purpose: visible comment paths stay short, while hidden background priming gets a longer window to make later comments fast.

Latency and caching

Fast path

The best-case path is parse commands, hit the menu cache, reuse a shared S3 URL, and patch the issue body. That avoids live stepping and avoids uploading a duplicate frame.

Logs you care about

  • render_done covers cache lookup or live Doom render work.
  • publish_done covers local or S3 frame publishing.
  • issue_patch_done covers GitHub issue body patch latency.
  • total_job_ms is the full queued job time after webhook acceptance.

Remaining bottlenecks are still external or unavoidable: GitHub patch time, S3 upload time for uncached frames, and image refresh latency on GitHub itself.

Operations

Recommended environment

  • GITHUB_TOKEN, GITHUB_OWNER, GITHUB_REPO, GITHUB_WEBHOOK_SECRET
  • PUBLIC_BASE_URL and DATABASE_URL
  • FRAME_STORE=s3 plus bucket and AWS credentials
  • DOOM_ENGINE=doomgeneric
  • Menu and boot frame cache enabled for low-latency startup states

Release checklist

  1. Build the Docker image so the DoomGeneric worker binary is rebuilt.
  2. Deploy with DOOM_ENGINE=doomgeneric.
  3. Open a fresh issue and walk through cached menu states.
  4. Confirm early logs show cache_hit=true.
  5. Confirm later logs show mode=step instead of repeated fallback.

Quick summary

What v4 changed

User-facing

  • Early menu states can resolve from cached frames.
  • One comment can batch multiple commands into one final frame update.
  • The visible path avoids long replay stalls when a live worker is ready.

Runtime

  • The DoomGeneric worker stays alive and advances in memory.
  • Frame publishing uses deterministic S3 object naming for shared menu frames.
  • Background sync coalesces to the newest target instead of draining stale work.
